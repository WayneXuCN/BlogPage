> A personal knowledge base that continuously maintains itself as materials grow.

## Table of contents

## 1. Motivation

Knowledge workers face a large volume of information every day: articles, papers, reports, meeting notes. How to manage this information is an old problem.

### Option 1: Store it

The most common approach is to use Obsidian, Notion, Bear, or Apple Notes to build a folder structure and store materials by category. Organizing feels good at the time. But months later, when you try to find a specific viewpoint, you often can’t remember which folder or which file it was in. Organizing didn’t create understanding — it only moved information around.

An even bigger difficulty is that knowledge itself changes. The same concept may be phrased differently across articles. Different sources may present contradictory descriptions. A method in one field might be reinvented in another. To discover these connections and contradictions you need to remember the contents of many documents simultaneously — beyond a human brain’s capacity.

### Option 2: RAG retrieval

RAG (Retrieval-Augmented Generation) solves the problem of not being able to find something. Store documents in a vector database, retrieve relevant passages when asked, and let an LLM synthesize an answer. But RAG starts from the raw documents each time rather than from an already-digested body of knowledge. It assumes the original text is the optimal vessel of knowledge, which is not true. Raw documents contain redundancy, bias, outdated information, and author-specific phrasing. Something answered last week has to be rediscovered this week. When two articles contradict each other, RAG will present both without proactively flagging the disagreement.

### A third approach

Andrej Karpathy proposed a different direction in his LLM Wiki gist:

> Let the LLM act like a team of Wikipedia editors that continuously maintain a persistent Wiki.

When new material arrives, the LLM reads and understands it, extracts concepts and entities, updates existing pages, flags contradictions, and maintains cross-references. Queries then search this Wiki rather than the raw source documents. The core insight is: LLMs excel at understanding, synthesizing, and generating text, not at on-the-fly reassembling of fragments. Move comprehension into the ingestion phase and let queries retrieve from an already-structured knowledge base. That improves both efficiency and accuracy.

I built my own LLM Wiki workflow inspired by Karpathy and made many customizations. Below I describe the architecture, workflows, technical details, and design philosophy — what I did and why.

## 2. Architecture

The system has three layers with clear separation of responsibilities:

```text
PersonalAutoWiki/
├── ObsidianRaw/          # Raw sources (read-only, curated by humans)
│   ├── 00_Inbox/         # Temporary inbox, awaiting triage
│   ├── 01_Projects/      # Active projects, action-oriented, time-sensitive
│   ├── 02_Areas/         # Areas of responsibility, require maintenance
│   ├── 03_Resources/     # Reusable reference materials ← only this is ingested by the LLM
│   └── 04_Archive/       # Historical records
├── Wiki/                 # Knowledge base (maintained entirely by the LLM)
│   ├── Index.md          # Auto-maintained index
│   ├── Log.md            # Machine-parsable operation log
│   ├── Concepts/         # Concept pages (methodologies, theoretical frameworks)
│   ├── Entities/         # Entity pages (people, organizations, projects)
│   ├── Sources/          # Summaries of original materials
│   └── Outputs/          # Query outputs (high-value analysis persisted back)
├── .claude/skills/       # Skills definitions (codified workflows)
├── AGENTS.md             # The LLM's "employee handbook" (operational rules)
└── CLAUDE.md             # Reference to AGENTS.md
```

### 2.1 Layer one: Raw sources

I organize raw materials using the PARA method. PARA divides information into Projects, Areas, Resources, and Archives, with an Inbox as a temporary intake.

These five classes have distinct natures:

| Folder    | Nature                        | Example                                    | Fit for Wiki?                          |
| --------- | ----------------------------- | ------------------------------------------ | -------------------------------------- |
| Inbox     | Temporary, needs triage       | Newly clipped webpages, notes to be sorted | ❌ No — unstable                       |
| Projects  | Action-oriented, time-limited | Notes for "write annual report"            | ❌ No — tasks, not evergreen knowledge |
| Areas     | Ongoing responsibilities      | Team management notes                      | ❌ No — personal and action-focused    |
| Resources | Reusable reference materials  | Papers, frameworks, industry reports       | ✅ Yes — target for ingestion          |
| Archive   | Completed or inactive items   | Old project histories                      | ❌ No — historical, low priority       |

Key rule: The LLM only ingests files from `Resources`.

This is my first customization to Karpathy’s idea. Projects and Areas are action-oriented and require human execution; they should not be automatically ingested. The Inbox needs human selection before moving to Resources. Archives may contain useful material but have lower priority. Without this constraint, project notes and personal logs would pollute the Wiki. Resources are defined as "reference material likely to be useful in the future." That matches the Wiki’s goal: build a reusable knowledge graph.

### 2.2 Layer two: the Wiki (knowledge layer)

This is the system’s core. The Wiki contains four page types with clear responsibilities:

#### 2.2.1 Sources

For every ingested raw material, create a summary page under `Wiki/Sources/` that includes:

- Title, author, and source of the original document
- A concise 1–3 sentence core summary
- Key concepts and extracted entities
- Cross-references to other Sources

Sources are the entry point to the knowledge layer. Raw documents are condensed into structured summaries; redundancy and noise are filtered out. Tracking is a unique feature of Sources pages: each Source frontmatter contains three tracking fields:

```yaml
---
title: Resilience
created: 2026-04-08
updated: 2026-04-08
type: source
tags: [systems theory, ecology, resilience research]
source_path: Academic/Research Notes/Theory/Terms/resilience.md
source_hash: a1b2c3d4
source_mtime: 2026-04-08T17:30:00Z
---
```

| Field          | Purpose                                                             |
| -------------- | ------------------------------------------------------------------- |
| `source_path`  | Locate the original file and detect moves or deletions              |
| `source_hash`  | SHA256 first 8 hex digits to detect content changes                 |
| `source_mtime` | File modification time for quick checks to avoid hashing everything |

These fields let the Wiki automatically detect changes in raw sources. When an original file is edited, the system compares the hash to detect differences and flags the Source for re-ingestion.

#### 2.2.2 Concepts

`Wiki/Concepts/` stores abstract concepts and methodologies such as "resilience," "percolation theory," or "prospect theory." Concept pages:

- Present definitions synthesized from multiple Sources
- Flag contradictions when sources disagree
- Evolve as new Sources are ingested
- List all Sources that mention the concept

Concepts act as abstract nodes in the knowledge graph. Single-source definitions are often incomplete; multiple sources together approximate a fuller understanding.

#### 2.2.3 Entities

`Wiki/Entities/` contains concrete entities: people, organizations, projects, tools, places. Unlike Concepts, Entities are concrete objects and include:

- Basic facts about the entity
- Links to related Concepts (what ideas the entity engages with)
- Links to Sources that mention the entity
- Relationships to other Entities (collaborations, hierarchies, competition)
- A timeline where applicable

Entities are the concrete nodes of the knowledge graph — they ground abstract concepts in real contexts.

#### 2.2.4 Outputs

`Wiki/Outputs/` holds high-value query results written back into the Wiki. When a query produces cross-source synthesis, comparison tables, or insightful analysis, the result can be saved here.

Save-back criteria: Not every query should be persisted. Save only results that are:

- Reviews: cross-source syntheses
- Comparisons: structured compare tables of concepts or methods
- Analyses: discoveries of hidden relationships or insights
- Conclusions: reasoned judgments worth preserving

Simple lookups ("What are the three dimensions of framework X?") shouldn’t be saved because the answer lives in Sources and Concepts already. Outputs are the synthesis layer: they are new knowledge produced by reasoning and merit persistence.

#### 2.2.5 Index.md and Log.md

`Index.md` is a directory of all pages organized by type with one-line summaries. It controls scale — without an index, browsing dozens or hundreds of pages is hard.

`Log.md` is the operation log in a machine-parsable format:

```markdown
## [2026-04-08] ingest | Resilience assessment

- **Source**: ObsidianRaw/03_Resources/resilience-framework.md
- **Impact**:
  - **Sources**: framework-summary.md
  - **Concepts**: resilience.md
  - **Entities**: some-institute.md
  - **Index.md**, **Log.md**
- **Note**: Added a system-performance definition of resilience
```

The format is designed to be queryable with Unix tools:

```bash
grep "^## \[" Log.md | tail -5        # last 5 records
grep "| ingest" Log.md                 # all ingest operations
grep "| contradiction" Log.md         # all contradiction flags
grep "resilience" Log.md              # all operations touching "resilience"
```

The log should be like git history for the Wiki. Without it, the Wiki is a black box; with it, every change is traceable and auditable.

### 2.3 Layer three: Schema and Skills (instructions for the LLM)

#### 2.3.1 AGENTS.md / CLAUDE.md

`AGENTS.md` and `CLAUDE.md` are operational manuals covering:

- Directory responsibilities
- Detailed steps for the four core workflows
- File formats, naming conventions, and referencing rules
- Impact analysis strategy
- Contradiction detection and handling
- Search tool invocation order and priorities
- Response style requirements

The LLM reads these files at the start of each session so it knows its role and rules. Prompts alone are ephemeral and can be truncated; a stable handbook keeps behavior consistent.

#### 2.3.2 Skills

AGENTS.md defines rules; `.claude/skills/` contains executable Skill modules. Each Skill encapsulates triggers, steps, output formats, and edge handling:

| Skill                    | Purpose          | Trigger                |
| ------------------------ | ---------------- | ---------------------- |
| ingesting-resources      | Ingest resources | New files in Resources |
| querying-wiki            | Answer queries   | User questions         |
| checking-wiki-health     | Health checks    | Periodic audits        |
| detecting-resources-sync | Sync detection   | File system changes    |

Each Skill contains a `reference/` folder with page templates, frontmatter formats, and checklists. Claude Code recognizes Skills automatically. Editing a Skill changes the LLM’s behavior without touching AGENTS.md.

## 3. Four core workflows

### 3.1 Workflow 1: Detecting Resources Sync

Detecting sync addresses a straightforward problem: how does the Wiki know that a raw source has changed?

Without a sync check, ingestion only runs when a human explicitly asks. Raw materials are frequently edited. Fixes, supplements, restructures. Without automated detection the Wiki drifts from sources.

Process:

```text
Step 1: Scan ObsidianRaw/03_Resources/ for files
        Record relative path, content hash (SHA256 first 8 hex), and mtime
        ↓
Step 2: Read all pages under Wiki/Sources/ and extract tracking fields
        (source_path, source_hash, source_mtime)
        ↓
Step 3: Three-way comparison
        ├─ New: In Resources but not in Sources → queued for ingestion
        ├─ Changed: same path but hash differs → queued for re-ingestion
        ├─ Deleted: in Sources but not in Resources → needs handling
        └─ Synchronized: path and hash identical → no action
        ↓
Step 4: Generate a sync status report
```

Output example:

```markdown
## Resources sync status [2026-04-08 10:30]

### New files (2)

| Path                   | Modified         | Size  |
| ---------------------- | ---------------- | ----- |
| `Academic/NewTopic.md` | 2026-04-08 09:00 | 2.5KB |

### Changed (1)

| Path                            | Old hash | New hash | Modified         |
| ------------------------------- | -------- | -------- | ---------------- |
| `Academic/Theory/resilience.md` | a1b2c3d4 | e5f6g7h8 | 2026-04-08 08:30 |

### Synchronized (43)

43 files are synchronized.
```

Design notes:

- Use `source_mtime` for quick filtering to avoid hashing everything each run
- SHA256 first 8 hex is usually enough to detect edits while keeping frontmatter compact
- Renames appear as "deleted + new" and require human confirmation

Sync detection is the Wiki’s change sensor. Without it the Wiki is a static snapshot; with it the Wiki stays live.

### 3.2 Workflow 2: Ingesting Resources

When sync detection finds new or changed files, ingestion runs. When I ask the system to "process this file," the LLM executes a multi-step comprehension → association → update workflow:

```text
Step 1: Parse the original file structure: headings, sections, lists, tables, code blocks
        ↓
Step 2: Create a summary page in Wiki/Sources/
        - 1–3 sentence core summary
        - key points organized by original structure
        - extract key concepts and entities
        - write tracking fields
        ↓
Step 3: Extract concepts
        ├─ If present in Concepts/ → update: add source and expand definition
        ├─ If absent → create new concept page
        └─ If definitions conflict with existing ones → flag contradiction
        ↓
Step 4: Extract entities
        ├─ If present in Entities/ → update: add relations and attributes
        ├─ If absent → create new entity page
        └─ Add relationships to other entities and concepts
        ↓
Step 5: Impact analysis
        ├─ Which Sources need cross-references updated?
        ├─ Which Concepts require revision?
        ├─ Which Entities’ relationships change?
        └─ Which contradictions should be flagged?
        ↓
Step 6: Update Wiki/Index.md
        ↓
Step 7: Record the operation in Wiki/Log.md
        ↓
Step 8: Run qmd update && qmd embed to refresh the search index
```

Step 5 — impact analysis — is the crucial part. New material rarely affects only one page. A paper on infrastructure resilience could:

- Define a model of resilience → update `Concepts/resilience.md`
- Mention a research institute → update/create `Entities/some-institute.md`
- Cite another ingested paper → create cross-references between Sources
- Provide a definition that conflicts with an existing one → flag a contradiction in `Concepts/resilience.md`

Without impact analysis ingestion degrades into isolated summaries. Each Source would exist alone and Concepts and Entities would not be updated; the knowledge graph wouldn’t grow. Impact analysis makes each ingestion a global update.

LLMs are naturally good at this: a human reader rarely remembers all related concepts, entities, and sources at once. The LLM holds the Wiki context and can perform systematic association.

### 3.3 Workflow 3: Querying the Wiki

When a user asks a question, the LLM follows this flow:

```text
Step 1: Use the qmd tool to search the Wiki
        ├─ Lexical matches (exact keywords)
        └─ Vector semantic matches (embedding-based)
        ↓
Step 2: Read the retrieved pages and synthesize an answer
        ↓
Step 3: Cite sources with [[wiki-links]]
        ↓
Step 4: Evaluate whether the result should be saved to Outputs/
```

The core difference from RAG is the retrieval target. RAG searches raw documents. Wiki querying searches a pre-digested knowledge layer. Results are already condensed, contradictions flagged, and definitions synthesized across documents. Entity relations are already established.

When a query produces high-value output (review, comparison, analysis, or conclusion), the LLM recommends saving it to `Wiki/Outputs/`. Final saving is a human decision, keeping Outputs high quality.

The Skill `reference/query_patterns.md` defines five common query patterns and best practices, including lex/vec construction, intent disambiguation, and limits.

Example concept-definition query:

```javascript
mcp__qmd__query({
  searches: [
    { type: "lex", query: "concept-name definition" },
    { type: "vec", query: "What is concept-name, its meaning and explanation" },
  ],
  intent: "concept-definition",
  limit: 3,
});
```

Example comparison query:

```javascript
mcp__qmd__query({
  searches: [
    { type: "lex", query: "conceptA conceptB comparison difference" },
    {
      type: "vec",
      query: "Differences and relationships between conceptA and conceptB",
    },
  ],
  intent: "concept-comparison",
  limit: 5,
});
```

### 3.4 Workflow 4: Health checks

As the Wiki grows, maintenance becomes essential: outdated pages, unflagged contradictions, orphan pages, and frequently mentioned concepts lacking a canonical concept page are common problems.

Health checks periodically scan the Wiki and produce a report:

```text
Check item        | What it detects                    | Fix suggestion        | Priority
------------------|-------------------------------------|-----------------------|--------
Unsynced files    | New/changed files in Resources/     | Run ingestion         | P0
Missing concepts  | Links that point to nonexistent concept pages | Create missing pages | P1
Contradictions    | Pages with contradiction flags     | Human review          | P1
Unindexed pages   | Pages not listed in Index.md       | Update Index.md       | P2
Orphan pages      | Pages with no inbound links        | Add links or delete   | P3
```

Health reports are structured:

```markdown
# Wiki health report

Date: 2026-04-08

## Stats

| Type          | Count   |
| ------------- | ------- |
| Concept pages | 54      |
| Entity pages  | 34      |
| Source pages  | 45      |
| Output pages  | 0       |
| **Total**     | **133** |

## Problems

### Missing concepts (2)

- [[Concepts/adaptive-governance]] — referenced by [[Sources/resilience-framework]] but missing
- [[Concepts/socio-ecological]] — referenced by [[Entities/some-institute]] but missing

### Contradictions (1)

- [[Concepts/resilience]] has a dimensional definition conflict with [[Sources/another-framework]]

## Suggested actions

1. Ingest pages to create Concepts/adaptive-governance and Concepts/socio-ecological
2. Human review the contradiction in resilience
```

When a Wiki has dozens of pages, humans cannot maintain every association manually. The LLM’s periodic scans surface structural issues so humans only need to resolve flagged exceptions.

## 4. Typical workflows

The four workflows can be combined into end-to-end usage patterns.

### Scenario 1: First-time setup

```text
detecting-resources-sync  →  list files to ingest
ingesting-resources       →  batch ingest
checking-wiki-health      →  verify results
```

Detect Resources for intake, ingest them in batch, then run a health check to validate completeness.

### Scenario 2: Ongoing maintenance

```text
Weekly: detecting-resources-sync  → find changes
As needed: ingesting-resources    → update Wiki
Weekly: checking-wiki-health     → keep healthy
```

Detect changes, process them, and validate. The whole loop completes in minutes.

### Scenario 3: Knowledge query

```text
User asks a question → querying-wiki → answer → evaluate for Output
```

Simple queries answer quickly. High-value analyses are recommended to be saved as Outputs.

## 5. Notes

### 5.1 Local search: qmd

The Wiki uses `qmd` as the local Markdown search engine.

Hybrid retrieval: lexical + vector

```javascript
mcp__qmd__query({
  searches: [
    { type: "lex", query: "critical infrastructure" }, // exact match
    { type: "vec", query: "How to assess infrastructure resilience" }, // semantic match
  ],
  intent: "academic research methods",
  limit: 5,
});
```

Different phrasings mean lexical search can miss results; pure vector search can return semantically related but off-topic results. Hybrid search balances precision and recall. `intent` helps disambiguate multi-meaning queries.

I chose `qmd` because all data stays local and it can be exposed to the LLM as an MCP tool. It supports path filters, document IDs, and glob patterns.

### 5.2 Contradiction flags

When an ingested Source conflicts with existing content, the LLM does not decide which is correct. It explicitly flags the contradiction in the page:

```markdown
> [!warning] Contradiction
> This page conflicts with [[Sources/another-framework]]:
>
> - This page’s view: resilience has three dimensions (engineering, ecological, social)
> - Other source’s view: resilience has five dimensions (adds economic and governance)
> - Possible reason: different application domains (natural hazards vs urban infrastructure)
```

Design principles:

1. The LLM is not the domain expert and should not unilaterally resolve contradictions. Flagging preserves transparency for human review.
2. Contradictions are often about scope, context, or framing rather than simple right/wrong. Keeping both perspectives is valuable.
3. A good contradiction flag attempts to analyze why the difference arises (different domain, different definition scope, time). That analysis itself is valuable.
4. Use Obsidian-style callouts `> [!warning]` so they render clearly in Obsidian.

### 5.3 Operation log

`Log.md` follows log best practices:

```markdown
## [2026-04-08] ingest | Infrastructure resilience framework

- **Source**: ObsidianRaw/03_Resources/resilience-framework.md
- **Impact**:
  - **Sources**: framework-summary.md
  - **Concepts**: resilience.md
  - **Entities**: some-institute.md
  - **Index.md**, **Log.md**
- **Note**: Added three-dimension definition of resilience
```

Principles:

1. Machine-parsable: each record starts with `## [date] type | title` so regex queries work
2. Traceable: each record lists sources and impacted pages to trace changes
3. Quantifiable: `grep | wc -l` can count operation frequencies
4. Readable: human-friendly format without extra tooling

The operation log is the Wiki’s git history — without it, the Wiki loses provenance.

### 5.4 Page templates

Each Skill’s `reference/` folder contains page templates to ensure consistent outputs:

- Sources template: summary → core points → key concepts → related entities → references
- Concepts template: definition → core features → related concepts → sources
- Entities template: overview → info table → related concepts → sources
- Outputs template: question → answer → sources → related concepts

Templates make pages predictable so humans know where to look and automated tools can parse them reliably.

## 6. Systematic customizations to Karpathy’s design

Karpathy’s gist described Raw Sources → Wiki → Schema and ingest/query/lint flows. I engineered enhancements:

| Dimension              | Karpathy                  | My customization                       | Why                   |
| ---------------------- | ------------------------- | -------------------------------------- | --------------------- |
| Ingestion scope        | unspecified               | only ingest Resources directory        | prevent noise         |
| Contradiction handling | mentioned                 | explicit flags + logs + diff analysis  | transparency          |
| Operation log          | simple timeline           | machine-parsable, grep-friendly log    | observability         |
| Search tool            | unspecified               | qmd hybrid search (lex + vec + intent) | precision + recall    |
| Output persistence     | mentioned                 | clear criteria + human decision        | quality control       |
| Health checks          | lint-like                 | extended checks with priorities        | system maintenance    |
| Collaboration rules    | informal                  | full AGENTS.md handbook                | reduce LLM randomness |
| Impact analysis        | not detailed              | explicit analysis step                 | global updates        |
| Page types             | Sources/Entities/Concepts | add Outputs for saved analyses         | capture synthesis     |
| Sync detection         | none                      | three-way tracking fields              | continuous sync       |
| Workflows              | conceptual                | Skills modularization with refs        | executable workflows  |

Karpathy’s gist is a design document; mine is an engineered runbook. AGENTS.md + Skills make the process executable. Changing a Skill changes behavior without editing the handbook.

## 7. Design philosophy

### 7.1 LLM as a knowledge bookkeeper

The tedious bookkeeping of knowledge maintenance is exactly what LLMs handle well:

- Update concept definitions when new materials arrive
- Flag contradictions between sources
- Convert a good query answer into a reusable page
- Periodically scan for stale or missing pages
- Track whether raw sources have changed

Humans focus on curating what to ingest, asking good questions, resolving contradictions, and deciding what to persist. LLMs handle repetitive, scale-limited bookkeeping.

### 7.2 Knowledge evolves

Notes assume knowledge is static; real knowledge evolves:

- New concepts may force revisions of old definitions
- New evidence can overturn prior conclusions
- Cross-domain connections create novel understanding
- A concept can mean different things in different contexts

The Wiki accepts evolution: contradictions coexist until human resolution; impact analysis updates related pages; sync detection tracks source edits; health checks surface issues.

### 7.3 Constraints create value

AGENTS.md and Skills enforce constraints: only ingest Resources, use consistent templates, prefer `[[wiki-links]]`, never let the LLM unilaterally resolve contradictions, require criteria for Outputs, and include tracking fields.

Constraints channel the LLM’s power into the right places. Without scope limits, the LLM ingests noise. Without formats, pages become inconsistent and hard to parse. Without persistence criteria, Outputs degenerate into low-value clutter.

### 7.4 Clear human/LLM division

| Task                       | Human | LLM |
| -------------------------- | ----- | --- |
| Curate sources             | ✅    | ❌  |
| Read & summarize           | ❌    | ✅  |
| Association analysis       | ❌    | ✅  |
| Change detection           | ❌    | ✅  |
| Contradiction adjudication | ✅    | ❌  |
| Ask questions              | ✅    | ❌  |
| Synthesize answers         | ❌    | ✅  |
| Quality gating for Outputs | ✅    | ❌  |
| Periodic maintenance       | ❌    | ✅  |
| Workflow definition        | ✅    | ❌  |

### 7.5 Composable, modifiable workflows

Skills let workflows be tools that compose:

- `detecting-resources-sync` can run standalone or inside `checking-wiki-health`
- `ingesting-resources` can be manual or triggered by sync detection
- `querying-wiki` can answer or produce Outputs
- `checking-wiki-health` can run post-ingest or periodically

Humans choose which tools to run and when.
